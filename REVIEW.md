# サービスレビュー（line-app-frontend）

実施日: 2026-03-12

## サマリー
- 主要な懸念は **認証/認可の実装がフロントエンド中心で、権限制御がクライアントコード依存に見える点** です。
- 特に、登録フローで利用するPINの扱いと、匿名クライアントでのユーザー一覧取得は、実運用では情報漏えいリスクになり得ます。
- また、現状は lint エラーが多数あり、品質ゲートが機能していません。

## 指摘事項

### 1) ハードコードされた登録PINフォールバック（重大: High）
- `LiffProvider` の登録処理で、`NEXT_PUBLIC_REGISTRATION_PIN` 未設定時に `@Result2020` がフォールバック値として使用されています。
- `NEXT_PUBLIC_` 変数およびクライアントバンドル内の文字列は利用者から参照可能であり、認証要素としては弱くなります。

対象:
- `src/components/LiffProvider.tsx` の `expectedPin` 決定ロジック。

推奨対応:
- PIN照合をクライアントで行わず、サーバー側（Edge Function / API Route / Supabase RPC）で実施。
- クライアント側には成功/失敗のみ返し、期待PIN値は一切公開しない。
- フォールバック固定値を廃止し、未設定時は登録処理自体を停止する。

### 2) 匿名クライアントでの全ユーザー取得（重大: High）
- `getAllUsers()` が `supabaseAnon`（`NEXT_PUBLIC_SUPABASE_ANON_KEY`）で `users` 全件を取得しています。
- コメント上も「RLS制限回避」が意図されており、RLS設計によっては匿名・一般ユーザーから過剰な情報取得が可能になります。

対象:
- `src/lib/api/admin.ts` の `supabaseAnon` 定義と `getAllUsers()`。

推奨対応:
- 管理系一覧取得はサーバー側経由に限定（サーバーで管理者判定後に問い合わせ）。
- Supabase側は `anon` から `users` テーブル直接SELECTを禁止し、必要最小限のSECURITY DEFINER RPCのみ公開。
- `phone_number` など個人情報カラムの返却範囲を役割ベースで分離。

### 3) 招待リンク処理における改ざん耐性不足（重要: Medium）
- `invite_store_id` / `link_id` をURLクエリから直接取得し、クライアントからDB更新処理に進む実装です。
- トークン署名・期限・1回限り利用などの検証が見当たらず、URL共有や推測時の悪用耐性が弱い構造です。

対象:
- `src/components/LiffProvider.tsx`（`link_id`, `invite_store_id` の利用）
- `src/lib/api/admin.ts`（`joinStore`）

推奨対応:
- 招待URLを `signed token (JWT/HMAC)` 化し、サーバーで有効性（署名・期限・対象ユーザー）を検証してから更新。
- `joinStore` はクライアント呼び出しで任意 userId を受け取らない設計に変更。

### 4) 管理画面アクセス制御のクライアント依存（重要: Medium）
- 管理画面のアクセス判定は `admin/layout.tsx` のクライアントロジック (`user.role === 'STAFF'`) が中心です。
- UIブロックだけでは防御にならず、バックエンド側ポリシー不備があると直接API呼び出しで操作できる恐れがあります。

対象:
- `src/app/admin/layout.tsx`
- `src/lib/api/admin.ts` の各更新API

推奨対応:
- 更新系処理はサーバー側エンドポイントに集約し、毎回ロール検証を実施。
- Supabase RLSで `role in ('PRESIDENT','EXECUTIVE','MANAGER','ADMIN')` などを厳密に定義。

### 5) 品質ゲート不全（中: Medium）
- `npm run lint` が多数エラーで失敗しています（`no-require-imports`, `no-explicit-any` など）。
- この状態では新規不具合の早期検出が難しく、運用上の回帰リスクが高いです。

対象:
- ルート配下の多数ファイル（特に運用スクリプト群）
- `eslint.config.mjs`

推奨対応:
- 本番対象外スクリプトを `globalIgnores` へ明示追加、または `scripts/` へ隔離し別lintルール化。
- `src/` 配下は `no-explicit-any` 解消を優先し、CIで lint 必須化。

## 優先度付き対応計画（提案）
1. **即時**: フォールバックPIN削除、登録検証をサーバー側へ移管。
2. **即時**: `anon` クライアントによる `users` 一覧取得を停止。
3. **短期**: 招待リンクの署名付きトークン化。
4. **短期**: 管理APIのサーバー集約 + RLS見直し。
5. **中期**: lint 対象整理と `any` 削減。

## 補足
- 本レビューは静的コード確認と lint 実行結果を基に作成しています。
- 実DBポリシー（RLS SQL）と運用設定を合わせて再評価すると、優先度の精緻化が可能です。

## 社内ツール運用（現状）でも起こりうる不都合
- **URLの転送・誤共有で権限外の店舗紐付けが起きる可能性**: `invite_store_id` / `link_id` がクエリベースのため、チャット誤送信や履歴共有で第三者が踏めると意図しない紐付けが発生し得ます。
- **内部者による情報閲覧範囲の過大化**: `getAllUsers()` が匿名キー系クライアントで実装されているため、RLS定義が弱い環境では「社内だから」で済まない個人情報閲覧が起こり得ます。
- **暫定PINの恒久運用化**: 固定フォールバックPINが残っていると、運用担当交代後も同じ認証要素が使われ続け、内部不正や退職者の再アクセスリスクが増えます。
- **UI制御と実データ制御の乖離**: 画面でのブロック（`user.role === 'STAFF'`）だけで安心すると、API/DB側制御漏れ時に直接操作できるリスクがあります。
- **障害時の原因切り分け遅延**: lint が常時失敗している状態は、変更起因の不具合をノイズに埋もれさせ、社内ツールでも復旧時間を伸ばします。

> 結論: 「社内利用のみ」でも、誤操作・誤共有・内部者リスク・運用劣化は現実的に発生します。外部攻撃だけでなく、**業務停止やデータ整合性崩れ**が主な不都合になります。

## 明日本番予定に対する修正可否（結論）
- **結論: 「全面的な理想修正」は難しいが、高リスクを潰す限定対応なら本日中に現実的**。
- 不具合なく進めるには、**コード大改修を避け、設定/ポリシー中心のホットフィックス**に寄せるのが安全。

### 本日中にやるべき最小ホットフィックス（推奨）
1. **登録PINの固定フォールバックを無効化**
   - `@Result2020` フォールバックを削除し、未設定時は登録不可メッセージへ。
2. **`users` テーブルの anon 直接参照をRLSで明示拒否**
   - 既存画面が壊れる可能性があるため、必要なら一時的に管理者専用RPCを追加。
3. **招待リンクを暫定的に失効短縮**
   - 署名付き実装が間に合わない場合、運用でURL有効期限を短くし、発行後の使い回しを禁止。
4. **本番前スモークテストを固定化**
   - ログイン→スタッフ閲覧→シフト編集→交通費登録→招待リンクの5導線のみでも必ず実施。

### 明日までに「安全に出す」ための判断基準（Go/No-Go）
- **Go**
  - 上記4項目を完了し、主要導線の手動確認で致命不具合なし。
  - DBバックアップ/ロールバック手順を事前に確認済み。
- **No-Go**
  - anon経由のユーザー一覧が残る、またはPINフォールバックが残る。
  - 招待リンク悪用時の停止手段（無効化手順）が未整備。

### リスクを増やさない実装方針
- 新規機能追加は止め、**既存処理の防御強化のみ**に限定。
- 変更は小さく（1PR 1目的）、各PRで手動検証項目を明記。
- デプロイ直後30分は監視担当を固定し、異常時は即ロールバック。

## 補足回答: 「社用電話番号の事前登録 + 認証のみ」で十分か？
- **結論: 単独では不十分です。**
- 理由は、電話番号は「本人確認の要素の1つ」に過ぎず、次の運用リスクを残すためです。
  1. **番号の再利用/使い回し**: 法人回線の再割当や端末貸与で、過去利用者と新利用者の境界が曖昧になります。
  2. **誤登録/誤入力耐性の弱さ**: 管理台帳の入力ミスで、正規ユーザーが弾かれる/別人が通る事故が起こり得ます。
  3. **なりすまし対策の不足**: 端末盗難・LINEアカウント乗っ取り時に、電話番号一致だけでは検知が困難です。
  4. **認可（何ができるか）を代替できない**: 電話番号認証は「誰か」の確認であり、「どこまで操作可能か」は別途ロール/RLSで制御が必要です。

### 明日リリース前の現実的な最低ライン
- 電話番号認証は維持しつつ、**以下を追加できれば実務上の安全性が大きく向上**します。
  1. **サーバー側で最終判定**（クライアント判定のみで通さない）。
  2. **ロール別RLSの強制**（閲覧・更新範囲をDBで制限）。
  3. **招待リンクの短命化/ワンタイム化**（流出時の被害縮小）。
  4. **監査ログ**（誰がいつ連携/更新したか）を最低限保存。

> 実務判断: 「社用電話番号の事前登録」は有効な防御ですが、**入口対策**に留まります。明日出すなら、入口対策に加えて「サーバー側認可」と「DB側制御」を最低限セットで入れるのが安全です。
